# 视觉分析平台 — 系统流程总结

> 更新时间：2026-04-01  
> 本文档描述当前系统各模块的完整数据流、交互关系和核心逻辑。

---

## 一、系统架构总览

```
┌─────────────┐        HTTP        ┌──────────────────┐       HTTP        ┌──────────────────────┐
│   Frontend   │  ───────────────→  │   vision-admin   │  ─────────────→  │  vision-inference     │
│  Vue 3 + TS  │  ←───────────────  │  Spring Boot 3.2 │  ←─────────────  │  Flask + YOLO (×N)   │
└─────────────┘                     └──────────────────┘                   └──────────────────────┘
                                           │                                       │
                                    ┌──────┴──────┐                         ┌──────┴──────┐
                                    │ PostgreSQL  │                         │ YOLO 模型文件 │
                                    │ + Flyway    │                         │ .pt / .onnx  │
                                    └─────────────┘                         └─────────────┘
                                           │
                                    ┌──────┴──────┐
                                    │ StorageService│
                                    │ 本地磁盘/MinIO│
                                    └─────────────┘
```

**关键设计：多节点推理**  
- vision-inference 可部署多个实例（GPU 服务器 / CPU 服务器各一套）
- 每个推理节点启动时向 vision-admin 注册，之后每 15 秒心跳上报
- 一个模型可同时部署到多个节点，通过 `model_node_deployment` 关联表管理

## 二、核心数据模型

### 2.1 model 表（模型元数据）

| 字段 | 说明 |
|------|------|
| id | UUID 主键 |
| name, version, businessTag | 基本信息 |
| taskType | detect / segment / classify / pose |
| **classNames** | 模型分类名称，逗号分隔（如 `person,car,bicycle`）— 由推理节点解析填充 |
| **numClasses** | 分类数量 |
| **parsedStatus** | `pending` → `parsed` → `failed`，标记模型元数据是否已解析 |
| confidenceThreshold, inputResolution | 推理参数 |
| modelPath | 模型文件存储路径 |

> **注意**：model 表不再存储 `status`（loaded/unloaded）、`device`、`nodeId`。这些信息拆分到了 `model_node_deployment` 表。

### 2.2 model_node_deployment 表（模型 ↔ 节点部署关联）

| 字段 | 说明 |
|------|------|
| id | UUID 主键 |
| model_id | 关联模型 |
| node_id | 关联推理节点 |
| device | cpu / cuda |
| device_name | 如 `NVIDIA GeForce RTX 3090` |
| status | loading / loaded / error |
| deployed_at | 部署时间 |
| **UNIQUE(model_id, node_id)** | 同一模型在同一节点只能部署一次 |

### 2.3 inference_node 表（推理节点注册信息）

| 字段 | 说明 |
|------|------|
| id | UUID（首次注册分配，之后持久化在推理节点本地） |
| nodeName, host, port | 节点访问信息 |
| status | online / offline |
| deviceType | cpu / cuda |
| gpuName, gpuCount, cpuInfo, memoryTotal | 硬件信息 |
| lastHeartbeat | 最后心跳时间 |

## 三、核心流程详解

### 3.1 推理节点注册与心跳

```
Python 节点启动
  │
  ├─→ registration.py: 收集硬件信息（CPU/GPU/内存）
  │
  ├─→ POST /api/v1/nodes/register
  │     {nodeId, nodeName, host, port, deviceType, gpuName, ...}
  │
  ├─→ Java NodeService.register()
  │     ├─ 按 host:port 幂等查重
  │     ├─ 创建或更新 inference_node 记录，status='online'
  │     └─ 发布 NodeOnlineEvent ──→ 触发待解析模型自动补全（见 3.3）
  │
  └─→ 启动心跳守护线程（每 15 秒）
        POST /api/v1/nodes/heartbeat
        {nodeId, loadedModels[], activeTasks[], systemLoad{cpu%, mem%, gpu%}}
```

### 3.2 模型上传与元数据解析

```
前端上传模型文件
  │
  ├─→ POST /api/v1/models/upload（multipart: file + name + version + taskType + ...）
  │
  ├─→ Java ModelService.uploadModel()
  │     ├─ 保存文件到 StorageService → 获得 modelPath
  │     ├─ 创建 Model 记录，parsedStatus = 'pending'
  │     └─ 异步调用 tryParseModelOnUpload()
  │
  ├─→ tryParseModelOnUpload()
  │     ├─ 通过 NodeRouter.selectAnyOnlineNode() 找一个在线节点
  │     │   └─ 如果没有在线节点 → 保持 pending，等节点上线后自动补全
  │     ├─ 调用 InferenceClient.parseModel(nodeId, fileBytes, filename)
  │     │
  │     └─→ Python POST /models/parse（multipart 文件）
  │           ├─ 临时加载 YOLO(model_path)
  │           ├─ 提取 model.names → class_names
  │           ├─ 提取 model.task → task_type
  │           ├─ 提取 model.overrides['imgsz'] → input_size
  │           ├─ 清理临时文件
  │           └─ 返回 {class_names, num_classes, task_type, input_size}
  │
  └─→ Java ModelService.applyParseResult()
        ├─ classNames = "person,car,bicycle"
        ├─ numClasses = 3
        ├─ taskType 和 inputResolution 被实际解析值覆盖
        └─ parsedStatus = 'parsed'
```

### 3.3 节点上线自动补全解析

```
NodeOnlineEvent 触发
  │
  └─→ ModelService.onNodeOnline()（@Async @EventListener）
        ├─ 查询所有 parsedStatus='pending' 的模型
        └─ 对每个 pending 模型：
             ├─ 获取模型文件本地路径
             ├─ 调用 inferenceClient.parseModelByPath(nodeId, modelPath)
             └─ 解析成功则 applyParseResult()，失败则 log.warn
```

**场景**：没有推理节点在线时上传模型 → `parsedStatus='pending'` → 之后推理节点注册上线 → 自动触发解析补全。

### 3.4 模型加载（部署到节点）

```
前端点击「加载到节点」→ 选择节点 + 设备（CPU/GPU）
  │
  ├─→ POST /api/v1/models/{id}/load?device=cuda&nodeId=xxx
  │
  ├─→ Java ModelService.loadModel()
  │     ├─ 检查该模型在目标节点是否已部署 → 已部署则报错
  │     ├─ 创建 ModelNodeDeployment 记录，status='loading'
  │     ├─ 若节点上没有模型文件：调用 inferenceClient.uploadModelFile()
  │     ├─ 调用 inferenceClient.loadModel(nodeId, modelId, remotePath, device)
  │     │     └─→ Python POST /models/load {model_id, model_path, device}
  │     │           └─ ModelManager 加载 YOLO 模型到内存
  │     ├─ 更新 deployment.status = 'loaded'
  │     └─ 若模型 parsedStatus='pending'，顺便解析元数据
  │
  └─→ 前端刷新 → 模型卡片显示「已部署 N 节点」+ 各节点设备信息
```

### 3.5 模型卸载

```
方式一：全部卸载（前端「全部卸载」按钮）
  POST /api/v1/models/{id}/unload
  → ModelService.unloadModelAll() → 遍历所有 deployment 逐个卸载 + 删除记录

方式二：单节点卸载（详情面板每个节点旁「卸载」按钮）
  POST /api/v1/models/{id}/unload?nodeId=xxx
  → ModelService.unloadModel(id, nodeId) → 调用 Python /models/unload → 删除该条记录
```

### 3.6 模型删除

```
前端点击删除
  │
  ├─ 检查 deployments 是否为空 → 不为空则提示「请先卸载」
  │
  └─→ DELETE /api/v1/models/{id}
        ├─ 再次检查无部署记录
        ├─ 删除文件存储
        └─ 删除 Model 记录（软删除）
```

### 3.7 节点故障处理

```
探测到节点离线（心跳超时等）
  │
  └─→ NodeFailoverService.handleNodeOffline(nodeId)
        ├─ 查询该节点所有 model_node_deployment 记录
        ├─ 逐条删除部署记录
        └─ 日志记录受影响的模型数量
```

### 3.8 模型测试（单张图片推理）

```
前端上传图片 → POST /api/v1/inference/test（multipart: image + modelId + confidence）
  │
  ├─→ InferenceService.testInference()
  │     ├─ 查找模型
  │     ├─ modelService.findLoadedDeployment(modelId) → 找到已加载的部署节点
  │     ├─ 保存图片到 StorageService
  │     ├─ inferenceClient.predict(nodeId, filePath, modelId, confidence, taskType)
  │     │     └─→ Python POST /predict → YOLO 推理 → 返回检测结果
  │     └─ 转换为前端格式返回
  │
  └─→ 返回 {taskType, inferenceTimeMs, objects[], classifications[]}
```

### 3.9 监测任务调度

```
TaskScheduler（每 30 秒扫描一次）
  │
  ├─→ 查询所有 status='running' 的监测任务
  │
  └─→ 对每个任务 processTask():
        ├─ ① 检查时间窗口（日期范围、星期、时段）
        ├─ ② 校验模型至少在一个节点上加载
        │     modelService.findLoadedDeployment(task.modelId)
        │     └─ 为空则 warn 跳过
        ├─ ③ 展开分组 → 获取摄像头列表
        ├─ ④ 解析抓图频率（如 "5min" → 300 秒）
        └─ ⑤ 遍历摄像头，按频率提交异步执行
              └─ TaskInferencePipeline.execute(task, camera, model)
```

### 3.10 任务级推理管道

```
TaskInferencePipeline.execute(task, camera, model)
  │
  ├─ ① 抓帧：FFmpeg 从摄像头 RTSP 流截取一帧 JPEG
  │
  ├─ ② 上传：保存到 StorageService
  │
  ├─ ③ 选择节点 + 推理：
  │     selectNode(task, model):
  │       优先级：task.nodeIds 指定节点 → modelService.findLoadedDeployment()
  │     inferenceClient.predict(nodeId, imageUrl, modelId, confidence, taskType)
  │
  ├─ ④ 保存推理记录（inference_record + detection）
  │
  ├─ ⑤ 更新任务统计：推理次数 +1
  │
  ├─ ⑥ 条件评估：TaskConditionEvaluator.evaluate(detections, task)
  │     匹配 alertTarget 中指定的分类 + alertConfidence 阈值 + alertFrames 连续帧数
  │
  └─ ⑦ 若触发告警：
        ├─ 创建 Alert 记录
        ├─ 更新推理记录告警状态
        ├─ 更新任务告警统计
        └─ callbackService.pushAlert()（WebSocket / 回调 URL）
```

### 3.11 监测任务启动校验

```
启动任务 POST /api/v1/tasks/{id}/start
  │
  └─→ MonitorTaskService.startTask()
        ├─ 校验模型存在
        ├─ 校验 modelService.findLoadedDeployment() 不为空（至少有一个节点部署）
        ├─ 校验分组下有摄像头
        └─ 设置 status='running'
```

## 四、前端交互流程

### 4.1 模型管理页面（ModelPage.vue）

| 区域 | 展示内容 |
|------|----------|
| 模型卡片列表 | 业务标签、版本、任务类型、**解析状态**标签、**已部署 N 节点**标签 |
| 部署节点列表（卡片内） | 每个节点名称 + 设备类型（CPU/GPU）标签 |
| 操作按钮 | 「加载到节点」（弹窗选节点+设备）、「全部卸载」、「参数」、「删除」 |
| 详情面板 — 基本信息 | 模型路径、创建时间、作者、**解析状态**、**分类数量** |
| 详情面板 — 模型分类名称 | 展示 `classNames` 数组为 Tag 列表 |
| 详情面板 — 部署节点 | 各节点行：节点名 + 设备 + 状态 + 单独「卸载」按钮 |
| 详情面板 — 动态参数 | 置信度阈值、最大并发数、推理分辨率（可编辑保存） |

### 4.2 监测任务页面（MonitorTaskPage.vue）

| 字段 | 交互方式 |
|------|----------|
| 模型选择 | 下拉列表，选中后自动加载该模型的 classNames |
| **检测目标（alertTarget）** | **多选下拉（n-select multiple）**，选项来自所选模型的 classNames，支持 tag 模式和搜索过滤 |
| 置信度阈值 | 数字输入 |
| 连续帧数 | 数字输入 |

> 检测目标不再是自由文本输入，而是从模型实际的分类名称中多选，避免手动输入导致的拼写错误和不匹配问题。

## 五、数据库迁移记录（V10）

V10 迁移脚本完成以下变更：

1. **model 表新增列**：`class_names TEXT`、`num_classes INT`、`parsed_status VARCHAR(20) DEFAULT 'pending'`

4. **删除旧列**：`model.status`、`model.device`、`model.device_name`、`model.node_id`

## 六、关键技术决策

| 决策 | 原因 |
|------|------|
| 模型部署状态拆分到关联表 | 一个模型可同时部署到多个节点（GPU Server + CPU Worker），单字段无法表达 |
| 模型上传时自动解析元数据 | 用户不必手动输入分类名，系统自动从 YOLO 模型文件中提取 |
| 允许无节点时上传（pending 状态） | 系统启动顺序不可控，不能要求推理节点先于后端就绪 |
| NodeOnlineEvent 事件驱动补全 | 松耦合，避免 NodeService ↔ ModelService 循环依赖 |
| 推理节点选择策略 | 加载时用负载评分（CPU 30% + Mem 20% + GPU 50%），推理时用部署表查找 |
| 心跳机制 + 运行时缓存 | Caffeine Cache TTL 60s 缓存节点运行时信息，心跳 15s 刷新 |
