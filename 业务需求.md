

# 视觉分析平台 — 业务需求文档

## 一、系统概述

视觉分析平台是一套面向工业/安防场景的后台管理系统，核心能力是：**接入摄像头 → 定时/事件抓图 → AI 模型推理 → 规则匹配 → 告警推送**。系统包含 5 个核心业务模块。

---

## 二、模块需求

### 模块 1：首页概览（Dashboard）

**目标**：为运维人员提供平台运行状态的全局视图，快速感知异常。

**功能点**：

| 编号 | 功能 | 说明 |
|------|------|------|
| D-01 | 指标卡片 | 展示 4 个核心指标：今日分析次数、今日告警数量、接入摄像头总数、启用 AI 摄像头数。每个指标需显示与昨日的环比变化（上升/下降/持平） |
| D-02 | 推理趋势图 | 近 7 天推理次数折线图，X 轴为日期，Y 轴为推理次数 |
| D-03 | 告警排行图 | 按业务线维度的告警数量水平条形图，降序排列 |
| D-04 | 实时告警列表 | 滚动展示最新告警，字段：告警时间、摄像头名称、告警类型、业务线。支持点击跳转告警详情 |
| D-05 | 快速操作入口 | 提供"新增摄像头"、"上传模型"、"新建规则"三个快捷按钮，点击后跳转对应功能页 |
| D-06 | 主题切换 | 支持深色/浅色两种主题模式 |

**数据来源**：聚合查询推理记录表、告警表、摄像头表。

### 模块 2：摄像头接入管理（Camera）

**目标**：管理平台接入的所有摄像头设备，配置抓图策略和 AI 分析参数。

**功能点**：

| 编号 | 功能 | 说明 |
|------|------|------|
| C-01 | 分组树 | 左侧树形结构，按业务线对摄像头分组，支持展开/折叠，点击分组过滤右侧列表 |
| C-02 | 摄像头列表 | 字段：摄像头 ID、名称、业务线、抓图频率（如"每 10 分钟一次"）、AI 启用状态（开关）、最后抓图时间、操作列 |
| C-03 | 搜索与筛选 | 按名称/ID 搜索，按业务线、状态筛选 |
| C-04 | 新增/编辑摄像头 | 抽屉式表单，含三个 Tab 页 |
| C-04a | — 基本信息 Tab | 名称（必填）、设备地址/RTSP 地址（必填）、位置描述 |
| C-04b | — 抓图配置 Tab | 触发方式：定时/事件（单选）；定时策略：固定间隔下拉 或 Cron 表达式输入；分辨率选择；是否存储原图（开关） |
| C-04c | — AI 分析配置 Tab | 启用 AI 分析（开关）、绑定业务线（下拉）、仅规则触发时抓图（开关） |
| C-05 | 行展开 | 列表每行可展开，显示该摄像头最近的抓图任务执行状态（成功/失败/时间） |
| C-06 | 删除摄像头 | 操作列提供删除功能，需二次确认 |
| C-07 | 查看抓图记录 | 操作列提供入口，查看该摄像头的历史抓图记录 |

**业务规则**：
- AI 启用状态变更时需校验是否已绑定业务线
- 设备地址需校验格式合法性（RTSP URL）

### 模块 3：模型配置管理（Model）

**目标**：管理 AI 推理模型的上传、版本、运行参数和生命周期（加载/卸载）。

**功能点**：

| 编号 | 功能 | 说明 |
|------|------|------|
| M-01 | 模型列表 | 卡片布局展示，每张卡片包含：模型名称、版本号、业务线标签、推理引擎类型（TensorRT/ONNX/ExecuTorch）、运行设备（GPU/CPU）、状态指示（已加载-绿色/未加载-灰色） |
| M-02 | 搜索与筛选 | 按模型名称搜索，按业务线、状态筛选 |
| M-03 | 加载/卸载模型 | 点击后向推理服务发送加载/卸载指令，状态实时更新 |
| M-04 | 模型详情展开 | 点击卡片展开，显示：模型文件路径、创建时间、版本变更记录、当前推理参数（置信度阈值、分辨率、并发数） |
| M-05 | 上传模型 | 向导式流程：① 选择模型文件（.pt/.engine/.pte）→ ② 填写名称、版本、业务线 → ③ 选择推理引擎 → ④ 上传完成，出现在列表中 |
| M-06 | 参数配置 | 弹窗编辑：置信度阈值（0~1 滑块）、推理分辨率（下拉）、设备选择（CPU/GPU 下拉）、最大并发数（数字输入） |
| M-07 | 替换模型 | 上传新版本文件替换当前模型，保留历史版本记录 |
| M-08 | 删除模型 | 需校验模型未被加载状态才允许删除，需二次确认 |

**业务规则**：
- 已加载状态的模型不允许删除，需先卸载
- 替换模型后自动创建版本记录
- 参数修改后，若模型已加载，需提示是否立即重载生效

### 模块 4：业务规则管理（Rule）

**目标**：以可视化方式配置 AI 推理结果的告警触发规则，支持热更新，无需重启服务。

**功能点**：

| 编号 | 功能 | 说明 |
|------|------|------|
| R-01 | 规则列表 | 左侧面板，展示所有规则，字段：规则名称、业务线、启用/禁用状态开关，支持搜索和筛选 |
| R-02 | 规则配置区 | 右侧面板，选中规则后展示完整配置，顶部有编辑/保存/测试按钮 |
| R-03 | 条件配置 | 可视化条件组编辑器，支持添加多个条件，条件类型包括：① 检测目标类别（下拉选择，如"泄漏"、"安全帽"）② 置信度阈值（滑块，如 > 0.7）③ 连续帧数（数字输入，如连续 3 帧）④ 多条件间支持 AND/OR 逻辑关系 |
| R-04 | 动作配置 | ① 告警等级：严重/警告/提示（单选）② 推送方式：WebSocket / HTTP 回调 / Kafka（复选）③ 回调地址输入框（选择 HTTP 回调时显示）④ 证据保存选项：图片 / 视频片段（复选） |
| R-05 | 生效时间配置 | 日期范围选择器（生效起止日期）、时间段设置（如 10:00-18:00）、生效星期选择 |
| R-06 | 规则测试 | 点击测试按钮，弹出模态框，上传样本图片，系统模拟推理 + 规则匹配，显示匹配结果和模拟告警内容 |
| R-07 | 发布规则 | 规则保存后点击"发布"，热更新生效，页面提示"规则已热更新，无需重启" |
| R-08 | 新建规则 | 从首页快捷入口或列表区新建，进入空白配置区填写 |

**业务规则**：
- 规则发布后即时生效，不需要重启任何服务
- 禁用规则后，已有未处理告警不受影响
- 条件组至少包含一个条件才允许发布

### 模块 5：推理历史查询（Inference）

**目标**：提供对历史推理记录的检索、查看、导出能力，便于回溯分析和审计。

**功能点**：

| 编号 | 功能 | 说明 |
|------|------|------|
| I-01 | 多维筛选 | 时间范围：快捷选项（今天/最近 7 天/本月）+ 自定义范围；业务线：多选下拉；摄像头 ID：搜索输入；告警类型：下拉选择；置信度范围：双向滑块 |
| I-02 | 结果列表 | 每条记录展示：缩略图（带检测框标注）、时间、摄像头名称、业务线、检测目标列表（如"泄漏 ×1，阀门 ×2"）、告警标记（红色标签，有告警时显示）、置信度平均值 |
| I-03 | 查看详情 | 侧滑抽屉展示：① 原图与标注图并排显示（支持缩放）② 推理结果 JSON（可复制）③ 关联的告警记录（如有）④ 下载图片、复制图片链接按钮 |
| I-04 | 数据导出 | 支持导出为 CSV 和 Excel 格式，导出内容根据当前筛选条件 |

**业务规则**：
- 列表默认按时间倒序排列
- 缩略图需展示 AI 标注的检测框和标签
- 推理结果 JSON 为完整的原始推理返回数据

## 四、核心业务流程

### 流程 1：抓图 → 推理 → 告警（主链路）

```
配置摄像头 & 设置抓图频率
        ↓
定时触发抓图（FFmpeg 从 RTSP 流截取一帧）
        ↓
图片存储（本地磁盘 / MinIO）
        ↓
调用 AI 推理服务（发送图片，返回检测结果）
        ↓
保存推理记录（含标注图、检测目标列表、置信度）
        ↓
规则引擎评估（匹配已启用的规则条件）
        ↓
命中规则 → 生成告警 → 实时推送前端（WebSocket）
```

### 流程 2：模型生命周期

```
上传模型文件 → 注册模型元数据 → 配置推理参数 → 加载到推理服务 → 运行推理 → 卸载/替换
```

### 流程 3：规则生命周期

```
新建规则 → 配置条件 & 动作 → 测试验证 → 发布热更新 → 运行时匹配 → 禁用/修改
```

## 五、各模块数据关系

```
摄像头(Camera) ──1:N──→ 推理记录(InferenceRecord) ──1:N──→ 检测目标(Detection)
                                    │
                                    └──0:N──→ 告警(Alert) ←── 规则(Rule)

模型(Model) ──1:N──→ 模型版本(ModelVersion)

摄像头分组(CameraGroup) ──1:N──→ 摄像头(Camera)
```

## 六、交付给 AI 开发的注意事项

1. **前后端接口契约**：后端 API 路径前缀统一为 `/api/v1`，响应格式统一为 `{ code, message, data }`
2. **分页约定**：请求参数 `page`（从 1 开始）、`size`（默认 20），响应 `{ items: [], total: number }`
3. **ID 生成**：所有主键使用 UUID（VARCHAR(36)）
4. **时间格式**：统一使用 ISO 8601（`yyyy-MM-dd'T'HH:mm:ss`）
5. **状态枚举值**：使用英文小写（如 `online`/`offline`/`loaded`/`unloaded`/`enabled`/`disabled`）
6. **文件存储路径约定**：`{类型}/{日期}/{文件名}`，如 `images/2026-03-29/cam001_abc123.jpg`
7. **技术架构详见** PROJECT.md，包含工程目录结构、数据库表结构、部署方案
