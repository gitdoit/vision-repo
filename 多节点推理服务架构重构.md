## Plan: 多节点推理服务架构重构

将推理服务从单机直连改为**注册发现 + 多节点调度**架构。推理服务启动后向管理服务注册并定时上报心跳与状态，管理服务维护节点注册表，实现模型加载/推理任务的**手动指定 + 自动调度**。单机部署时走同一套流程（本地推理服务自动注册），无需维护两套逻辑。

### 架构变更

```
当前：Java ── 固定地址 http://localhost:5000 ──→ 1个 Python 服务

目标：                    ┌── Node A (GPU, 192.168.1.10:5000)
      Java (vision-admin) ◀── 注册/心跳 ──┤
          │                  ├── Node B (GPU, 192.168.1.11:5000)
          │                  └── Node C (CPU, 192.168.1.12:5000)
          │
          ├── 节点注册表 (DB + Caffeine缓存运行时数据)
          ├── 节点路由器 (选择节点执行推理)
          └── 模型下载URL分发 (Java → 推理节点自行下载)
```

核心交互：
1. **注册**：推理服务启动 → `POST /api/v1/nodes/register` → 管理服务记录节点
2. **心跳**：每 15s → `POST /api/v1/nodes/heartbeat` → 携带已加载模型、活跃任务、系统负载
3. **健康检查**：管理服务每 30s 检查心跳超时 → 标记离线 → 自动迁移任务
4. **调度**：推理请求 → NodeRouter 选择节点 → 转发

### Decisions

| 决策 | 理由 |
|---|---|
| 推理服务主动注册（而非管理服务扫描） | 推理服务知道自己的地址，避免轮询探测 |
| 心跳 15s，超时 60s | 平衡实时性与网络抖动容忍 |
| 运行时信息存 Caffeine 不存 DB | 高频瞬态数据不适合持久化 |
| 模型文件由推理节点自行下载（Java 传 download_url） | 避免 Java 读文件再上传的冗余传输；推理节点本地持久化模型文件，重启可恢复 |
| 单机多机统一注册流程 | 避免两套代码路径 |
| 图片传输 URL + Base64 都支持 | 共享存储用 URL，跨网络用 Base64 兜底 |
| 不引入 Nacos/Eureka | 项目规模不大，自研轻量注册即可 |
| 每个 GPU 一个推理服务实例 | 简化管理，Docker 天然支持 |

### Further Considerations

1. **ADVERTISE_HOST**：Docker 网络用容器名，物理机需配置环境变量 `ADVERTISE_HOST` 指定外部可达 IP。建议两者都支持——有环境变量就用，否则自动检测本机 IP。
2. **模型文件传输效率**：YOLO 模型通常 6-150MB，HTTP multipart 推送可接受。V1 不做断点续传，后续如有需要再加。
3. **多 GPU 节点策略**：建议每个 GPU 启动一个推理服务容器实例，通过 `CUDA_VISIBLE_DEVICES` 隔离，容器层面管理比应用内管理更简洁。
