# 监测任务（MonitorTask）模块 — 设计与实现文档

> 最后更新：2026-03-31

## 一、模块概述

**监测任务**是视觉分析平台的核心业务编排单元，将**摄像头分组 + AI 模型 + 告警条件 + 回调推送**绑定为一个可调度、可启停的任务实体。

典型业务场景：城市内涝检测 — 创建一个监测任务，关联"易涝路段"摄像头分组和"积水检测"YOLO 模型，每 5 分钟抓帧推理，连续 3 帧检出积水且置信度 ≥ 0.6 时，通过 HTTP 回调推送告警到城市管理平台。

### 核心设计决策

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 任务与规则关系 | 嵌入式 | 告警条件直接嵌入任务，不复用 Rule 模块 |
| 任务与模型关系 | 一对一 | V1 一个任务绑定一个模型 |
| 回调方式 | HTTP + WebSocket | V1 仅支持 HTTP 回调和 WebSocket，MQTT/MQ 后续扩展 |
| 节点调度 | 可选指定 | 可指定节点列表，不指定则使用模型所在节点 |
| 告警抑制 | 不抑制 | 每次条件命中都推送 |
| 任务状态 | 三态 | stopped / running / error |

## 二、数据模型

### 2.1 数据库表：`monitor_task`

迁移脚本：`V9__add_monitor_task.sql`

```sql
CREATE TABLE vision.monitor_task (
    id                   VARCHAR(36) PRIMARY KEY,
    name                 VARCHAR(200) NOT NULL,          -- 任务名称
    description          TEXT,                            -- 任务描述
    business_line        VARCHAR(100),                    -- 业务线标签（城市内涝、交通管理等）
    group_id             VARCHAR(36) REFERENCES camera_group(id),  -- 关联摄像头分组
    model_id             VARCHAR(36) REFERENCES model(id),         -- 关联AI模型
    status               VARCHAR(20) DEFAULT 'stopped',   -- stopped / running / error

    -- 调度配置
    capture_frequency    VARCHAR(50) DEFAULT '5min',      -- 抓图频率（覆盖摄像头配置）
    schedule_start_time  VARCHAR(10),                     -- 生效时段起始 "08:00"
    schedule_end_time    VARCHAR(10),                     -- 生效时段结束 "22:00"
    schedule_weekdays    VARCHAR(20),                     -- 生效星期 "1,2,3,4,5"
    effective_start      DATE,                            -- 生效日期起
    effective_end        DATE,                            -- 生效日期止

    -- 告警条件（嵌入式）
    alert_target         VARCHAR(500),                    -- 检测目标类别，逗号分隔
    alert_confidence     DECIMAL(3,2) DEFAULT 0.50,       -- 置信度阈值
    alert_frames         INT DEFAULT 1,                   -- 连续帧数阈值

    -- 告警推送
    alert_level          VARCHAR(20) DEFAULT 'warning',   -- 告警级别 severe/warning/info
    push_methods         VARCHAR(200),                    -- 推送方式 "http_callback,websocket"
    callback_url         VARCHAR(1000),                   -- HTTP 回调地址
    callback_headers     JSONB,                           -- 自定义请求头

    -- 节点策略
    node_ids             VARCHAR(500),                    -- 指定节点列表，空=自动

    -- 统计
    total_inference      BIGINT DEFAULT 0,                -- 累计推理次数
    total_alert          BIGINT DEFAULT 0,                -- 累计告警次数
    last_inference_time  TIMESTAMP,                       -- 最后推理时间
    last_alert_time      TIMESTAMP,                       -- 最后告警时间

    created_at           TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at           TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    deleted              INT DEFAULT 0                    -- 软删除
);
```

同时为 `inference_record` 和 `alert` 表新增 `task_id` 列，关联推理记录和告警到具体任务。

### 2.2 实体关系

```
MonitorTask 1───* InferenceRecord    （一个任务产生多条推理记录）
MonitorTask 1───* Alert              （一个任务产生多条告警）
MonitorTask *───1 CameraGroup        （一个任务关联一个分组）
MonitorTask *───1 Model              （一个任务关联一个模型）
CameraGroup 1───* Camera             （一个分组包含多个摄像头）
```

## 三、API 接口

基础路径：`/api/v1/monitor-tasks`

| 方法 | 路径 | 说明 | 请求体 |
|------|------|------|--------|
| GET | `/` | 分页查询任务列表 | Query: `page`, `size`, `status`, `businessLine`, `modelId` |
| GET | `/{id}` | 查询任务详情 | - |
| POST | `/` | 创建监测任务 | `MonitorTaskCreateDTO` |
| PUT | `/{id}` | 更新监测任务（运行中不可编辑） | `MonitorTaskCreateDTO` |
| DELETE | `/{id}` | 删除监测任务（运行中不可删除） | - |
| POST | `/{id}/start` | 启动任务 | - |
| POST | `/{id}/stop` | 停止任务 | - |

### 3.1 创建/更新请求体

```json
{
  "name": "城市内涝监测-主干道",
  "description": "监测主干道积水情况",
  "businessLine": "城市内涝",
  "groupId": "分组UUID",
  "modelId": "模型UUID",
  "captureFrequency": "5min",
  "scheduleStartTime": "06:00",
  "scheduleEndTime": "22:00",
  "scheduleWeekdays": "1,2,3,4,5,6,7",
  "effectiveStart": "2026-04-01",
  "effectiveEnd": "2026-12-31",
  "alertTarget": "waterlogging,flooding",
  "alertConfidence": 0.60,
  "alertFrames": 3,
  "alertLevel": "severe",
  "pushMethods": "http_callback,websocket",
  "callbackUrl": "https://city-platform.example.com/api/alert",
  "callbackHeaders": "{\"Authorization\": \"Bearer xxx\"}",
  "nodeIds": "node-1,node-2"
}
```

**校验规则：**
- `name`、`groupId`、`modelId` 为必填（`@NotBlank`）
- 启动前校验：模型必须已加载（`status=loaded`）、分组下必须有摄像头
- 运行中的任务不允许编辑和删除

### 3.2 响应体 VO

响应在请求体基础上额外包含：`groupName`、`modelName`（展示用）、`totalInference`、`totalAlert`、`lastInferenceTime`、`lastAlertTime`、`createdAt`。

## 四、核心流程

### 4.1 调度流程

```
TaskScheduler (@Scheduled fixedDelay=30s)
  │
  ├── 1. 查询所有 status='running' 的 MonitorTask
  │
  ├── 2. 逐任务检查时间窗口
  │     ├── effectiveStart / effectiveEnd（日期范围）
  │     ├── scheduleWeekdays（星期过滤）
  │     └── scheduleStartTime / scheduleEndTime（时段过滤，支持跨午夜）
  │
  ├── 3. 校验模型仍然已加载
  │
  ├── 4. 展开分组 → 摄像头 ID 列表（通过 camera_group_mapping）
  │
  ├── 5. 按任务 captureFrequency 检查每个摄像头是否到期
  │     └── 使用 ConcurrentHashMap<taskId:cameraId, lastDispatchTime> 跟踪
  │
  └── 6. 到期 → 异步提交到 captureExecutor 执行 TaskInferencePipeline
```

### 4.2 推理管道流程

```
TaskInferencePipeline.execute(task, camera, model)
  │
  ├── 1. FFmpeg 抓帧 → 临时 JPEG 文件
  │
  ├── 2. StorageService 上传到存储（本地/MinIO）
  │
  ├── 3. 选择推理节点
  │     ├── 优先：任务指定节点列表（task.nodeIds）
  │     └── 兜底：模型所在节点（model.nodeId）
  │
  ├── 4. InferenceClient.predict() → Python 推理服务
  │
  ├── 5. 保存 InferenceRecord（含 task_id）
  │
  ├── 6. 保存 Detection 列表
  │
  ├── 7. 原子递增 task.totalInference
  │
  ├── 8. TaskConditionEvaluator 评估告警条件
  │     ├── 过滤目标类别白名单（alertTarget）
  │     ├── 过滤置信度阈值（alertConfidence）
  │     └── 连续帧数判断（alertFrames，Caffeine 缓存计数）
  │
  ├── 9. 命中 → 创建 Alert 记录（含 task_id）
  │
  ├── 10. 原子递增 task.totalAlert
  │
  └── 11. TaskCallbackService 推送告警
        ├── WebSocket → AlertPushService → /topic/alerts
        └── HTTP → POST callbackUrl（5s连接/10s读超时，失败重试2次）
```

### 4.3 告警条件评估逻辑

```
TaskConditionEvaluator.evaluate(detections, task, cameraId)
  │
  ├── 1. 推理结果为空 → 重置连续帧计数 → 返回空（不告警）
  │
  ├── 2. 解析目标白名单（alertTarget 逗号分隔）
  │     └── 空白 = 匹配所有类别
  │
  ├── 3. 过滤：label ∈ 白名单 且 confidence ≥ 阈值
  │     └── 无匹配 → 重置帧计数 → 返回空
  │
  ├── 4. alertFrames = 1 → 直接命中
  │
  └── 5. alertFrames > 1 → Caffeine 缓存计数
        ├── key = "taskId:cameraId"
        ├── 每次匹配 count++
        ├── count ≥ alertFrames → 命中，重置计数（防连续触发）
        └── TTL 10 分钟自动过期（场景变化重置）
```

## 五、回调报文格式

HTTP 回调 POST 到 `callbackUrl` 的固定 JSON 结构：

```json
{
  "alertId": "uuid",
  "alertLevel": "severe",
  "alertType": "waterlogging",
  "alertTime": "2026-03-31T10:24:05",
  "captureTime": "2026-03-31T10:24:00",
  "triggerCondition": "任务[城市内涝监测] 检测到: waterlogging(conf=0.85)",
  "task": {
    "taskId": "uuid",
    "taskName": "城市内涝监测-主干道",
    "businessLine": "城市内涝"
  },
  "camera": {
    "cameraId": "uuid",
    "cameraName": "主干道01号",
    "streamId": "channel-01",
    "location": "XX路与YY路交叉口"
  },
  "detection": {
    "modelName": "YOLOv8-Waterlogging-v2",
    "inferenceTimeMs": 45,
    "targets": [
      {
        "className": "waterlogging",
        "confidence": 0.85,
        "bbox": "120,80,320,450"
      }
    ]
  },
  "evidence": {
    "imageUrl": "https://storage.example.com/captures/2026-03-31/cam001_abc.jpg",
    "annotatedImageUrl": null
  }
}
```

**回调机制：**
- 超时设置：连接 5 秒 / 读取 10 秒
- 失败重试 2 次，递增等待（1s, 2s）
- 支持自定义请求头（`callbackHeaders` JSONB，如 Authorization）
- 异步执行（`@Async("captureExecutor")`），不阻塞推理管道

## 七、前端页面功能

`MonitorTaskPage.vue` 提供以下功能：

- **任务卡片列表**：展示名称、状态指示灯、分组名、模型名、抓图频率、告警级别、推理/告警统计
- **筛选**：按状态筛选、按名称搜索
- **创建/编辑弹窗**：
  - 基本信息：名称、描述、业务线
  - 关联配置：摄像头分组（下拉）、AI 模型（下拉）、抓图频率（预设选项）
  - 告警条件：检测目标、置信度阈值、连续帧数、告警级别
  - 推送配置：推送方式多选、HTTP 回调地址
  - 调度配置：生效时段、生效星期
- **操作按钮**：启动 / 停止 / 编辑 / 删除（删除需确认）

## 八、配置与依赖

### 线程池

复用已有 `captureExecutor`（core=5, max=20, queue=100），调度和推理管道在同一线程池异步执行。

### 依赖模块

```
MonitorTask 模块依赖：
├── camera 模块：CameraMapper, CameraGroupMapper, CameraGroupMappingMapper
├── model 模块：ModelMapper, InferenceClient
├── node 模块：NodeRouter
├── inference 模块：InferenceMapper, DetectionMapper
├── alert 模块：AlertMapper, AlertPushService
├── storage 模块：StorageService
├── common 模块：FFmpegUtil, IdUtil, BizException
└── 外部：Caffeine Cache（条件评估帧计数）
```

### 频率格式

支持格式：`30s` / `1min` / `5min` / `10min` / `30min` / `1h`，默认 `5min`（300 秒）。
